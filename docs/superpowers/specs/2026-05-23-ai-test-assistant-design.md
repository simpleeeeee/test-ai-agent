# AI 测试助手 Design Spec

## 目标

构建一个面向公司内部测试人员的桌面端 AI 测试应用。测试人员使用中文自然语言输入测试目标，例如“测试订单模块功能”，应用由 AI Agent 解析意图、生成测试计划，并通过预先接入的 MCP 服务调用测试工具完成执行、取证和缺陷草稿生成。

第一版重点是交互闭环和可审计过程，不做复杂测试管理平台。

## 产品定位

产品名称：AI 测试助手

使用者：公司内部测试人员，母语为中文。

核心场景：

- 测试人员输入一句自然语言测试目标。
- AI 识别模块、范围和测试意图。
- AI 生成简短测试计划，等待测试人员确认。
- AI 通过 MCP 服务调用浏览器、接口、数据库、截图、缺陷系统等工具。
- 应用展示执行过程、工具调用、授权请求、证据和缺陷草稿。

设计风格参考 Claude Desktop 的简洁对话工作台：低噪声、少面板、中文优先、以自然语言交互为中心。交互骨架要严格对齐 Claude Desktop：左侧会话列表，中央消息流，底部输入框，流式输出，工具调用内嵌在消息流里，不做仪表盘式布局。所有 UI 样式也要按 Claude Desktop 的气质统一，包括消息、边栏、输入框、按钮、状态、空状态、加载和工具事件。

## 非目标

第一版不做以下内容：

- 完整测试用例管理系统。
- 复杂测试套件编排平台。
- 多团队权限体系。
- 复杂数据看板。
- 大规模报告中心。
- 类 IDE 的多面板调试器。

这些能力可以在核心交互稳定后增量扩展。

## 关键用户流程

### 流程 1：自然语言发起测试

1. 测试人员打开应用。
2. 在输入框输入“测试订单模块功能”。
3. AI 返回识别结果：“已识别为订单模块回归测试”。
4. AI 生成测试计划。
5. 测试人员点击“开始执行”或“调整计划”。
6. AI 开始调用 MCP 工具执行测试。
7. 应用实时展示执行状态和关键证据。
8. 如果发现失败，AI 生成缺陷草稿。

### 流程 2：执行中授权

1. AI 执行到需要敏感工具时，例如查询订单数据库。
2. 应用在对话流中展示授权请求。
3. 测试人员选择“允许”或“拒绝”。
4. 允许后继续执行，拒绝后 AI 改用可用证据或提示无法完成该步骤。

### 流程 3：失败后生成缺陷

1. 某个步骤失败。
2. AI 汇总失败步骤、实际结果、预期结果和证据。
3. 右侧详情抽屉展示缺陷草稿。
4. 测试人员点击“生成缺陷草稿”。
5. 系统通过 MCP 缺陷服务创建草稿或复制复现步骤。

## 信息架构

### 左侧栏

左侧栏保持窄而简单，负责会话和基础入口。它的结构和 Claude Desktop 一致，以会话导航为主，而不是项目导航或分析看板。

固定入口：

- 新建测试
- 最近会话
- 设置

会话示例：

- 订单模块测试
- 支付回归
- 优惠券异常

左侧栏不承载复杂筛选器和统计看板。

左侧栏行为要求：

- 顶部固定一个“新建测试”入口。
- 最近会话可直接切换。
- 会话列表支持命名、重命名、删除和搜索。
- 选中会话后，中心区域立即切换到对应的消息流。
- 测试套件、MCP 工具、执行记录不作为主导航常驻项，它们通过命令入口、会话详情或设置页进入。

### 主工作区

主工作区是核心。它表现为 AI 对话工作台，但内容结构化展示测试过程。视觉和交互都要像 Claude Desktop，而不是企业控制台。

主要区域：

- 用户测试目标消息。
- AI 意图识别结果。
- AI 测试计划卡片。
- 执行过程流。
- 工具调用摘要。
- 授权请求。
- 底部中文输入框。

主工作区不使用大量聊天气泡。用户消息可以是简洁消息块，AI 输出以计划卡、执行行和结果块呈现。

主工作区交互要求：

- AI 输出必须支持流式生成，逐段出现，像 Claude Desktop 一样边思考边写。
- 工具调用必须嵌入到对应的 AI 消息下方，作为可折叠的事件行，而不是脱离上下文的独立面板。
- 计划确认、授权请求、失败提示都应尽量以内嵌消息块或轻量卡片出现。
- 用户可以在消息级别执行复制、重新运行、编辑并重发。
- 当 AI 正在生成时，输入区显示停止生成按钮。

### 详情抽屉

右侧详情抽屉不是常驻面板，默认隐藏，只在用户点选某条工具调用、某个证据或某个缺陷草稿时从右侧滑出。

内容：

- 当前项目。
- 当前环境。
- 当前 Agent。
- 已启用 MCP 工具数量。
- 证据缩略图。
- 缺陷草稿入口。

详情抽屉不做常驻数据看板，不抢占主对话区的视觉权重。

## Claude Desktop 式交互原则

这一版不是“像 Claude Desktop 一点”，而是把 Claude Desktop 的交互骨架作为默认模式。

必须遵守的交互规则：

- 中央区域始终是一条纵向消息流。
- 左侧永远是会话列表，不能变成项目首页。
- 用户发言和 AI 回复都按消息单位展示，消息之间有明确留白和顺序。
- AI 正在生成时，页面显示流式输出，而不是等待一个完整结果再一次性渲染。
- 工具调用、授权请求、失败提示都嵌在同一条消息上下文里。
- 不显示常驻的多面板仪表盘，不显示固定的分析区域。
- 右侧内容必须是按需打开的详情抽屉，不参与日常浏览。
- 输入框固定在底部，Enter 发送，Shift+Enter 换行，生成中显示停止按钮。
- 用户可以对消息复制、重新运行、编辑并重发。
- 行为层面更像聊天对话工作台，而不是测试管理系统。

## 视觉设计

整体风格要完全对齐 Claude Desktop 的界面气质：安静、克制、低对比、信息优先。界面看起来像一个会话工具，而不是一个后台系统。

### 色彩与层次

- 背景使用暖灰、米白或偏石色的浅底，不使用高饱和纯白。
- 主内容区采用白色或接近白色的浅面板。
- 页面只保留一到两个强调色，优先用于发送按钮、当前状态和少量关键操作。
- 文本层级使用深灰、次级灰、禁用灰，不使用花哨的渐变字色。
- 边框和分隔线必须细、轻、低存在感。
- 失败、警告、成功只作为状态色出现，不作为大面积装饰色。

### 侧栏样式

- 左侧会话栏要像 Claude Desktop 一样窄、稳、克制。
- 会话项是文本优先，不做大卡片。
- 选中态使用轻微底色变化和细边框，不使用强烈高亮。
- 悬浮态只做轻微背景变化和指针反馈。
- 顶部的新建入口使用简洁按钮，不做营销式 CTA。

### 消息样式

- 消息流采用整行区块式布局，不使用 Messenger 风格的大圆角气泡。
- 用户消息和 AI 消息通过缩进、背景层次和标题行区分，而不是靠夸张颜色区分。
- AI 回复可包含标题、正文、计划卡、工具事件和证据引用，但整体视觉必须统一。
- 消息卡片圆角保持克制，边距和内边距要接近 Claude Desktop 的密度。
- 代码块、命令块、日志块使用等宽字体和轻边框，不做强装饰。

### 输入框样式

- 输入框固定在底部，横向占满可用宽度。
- 输入区要像 Claude Desktop 一样是一个平静的创作区，而不是搜索框。
- 输入框使用浅底、细边框和足够的垂直内边距。
- 发送按钮只在可提交时突出显示，未可提交时保持弱化。
- 生成中显示停止按钮，位置和存在感要和 Claude Desktop 的行为一致。
- 附件、快捷命令、语音或工具图标保持低噪声风格，图标统一线性或轻实心，不使用夸张填充。

### 按钮与标签

- 主按钮使用低饱和、实心或轻强调样式，避免大块高对比色。
- 次按钮使用细边框或文本按钮，不堆叠过多视觉重量。
- 标签、状态 chip、工具名胶囊都应轻量、窄、高信息密度。
- 不使用大型圆角胶囊承载过长文案。
- 不使用过度拟物的阴影和立体感。

### 加载与空状态

- 生成中状态要像 Claude Desktop 一样，优先使用流式文本、打字感和轻量占位，而不是大 spinner。
- 空会话状态应简洁，只有一句引导文案和一个主操作入口。
- 无结果、无证据、无缺陷时只显示轻量提示，不做大插画。
- 骨架屏和加载提示必须和整体界面语言一致，不能像仪表盘组件。

### 字体与排版

- 中文优先，字体要有现代无衬线的轻盈感。
- 标题层级克制，不用夸张大标题。
- 消息正文、执行状态和按钮文案都要短、准、稳。
- 行高与间距保持舒展，但不能松散到像营销页。
- 代码、工具名和事件标识使用等宽字体或等宽感样式，便于扫描。

### 动效

- 仅使用少量、明确的动效。
- 消息流加载、折叠展开、详情抽屉滑出、生成中状态转换要有平滑过渡。
- 动效必须服务于“正在思考、正在执行、正在确认”的节奏，不做炫技。
- 不使用花哨弹跳、旋转装饰或过强的微交互。

## 交互逻辑

### 输入提交

当用户点击“发送”或按下 Enter：

1. 创建一条用户任务消息。
2. 将文本发送给 Agent。
3. Agent 返回意图识别结果。
4. Agent 返回测试计划。
5. UI 进入“等待确认”状态。

Shift+Enter 用于换行。

### 计划确认

计划生成后，用户可以：

- 点击“开始执行”：进入执行状态。
- 点击“调整计划”：输入修改要求，Agent 重新生成计划。

默认不自动执行，避免 AI 直接调用有副作用的测试工具。

### MCP 工具调用展示

每次 MCP 工具调用展示一行摘要：

- 工具名称，例如 `mcp-order.createOrder`。
- 中文动作说明，例如“创建测试订单”。
- 状态：待执行、执行中、已完成、失败、待授权。
- 简短输入输出摘要。

工具调用详情默认折叠，点击后展开。

### 授权策略

以下工具调用需要用户确认：

- 数据库查询。
- 数据写入。
- 创建缺陷。
- 执行会改变业务状态的操作。
- 访问敏感用户数据的操作。

授权选项：

- 允许：仅允许本次调用。
- 拒绝：跳过该调用并让 AI 解释影响。

第一版不提供“始终允许”，降低误操作风险。

### 执行状态

会话级状态：

- 空闲。
- 正在生成计划。
- 等待确认。
- 正在执行。
- 等待授权。
- 已完成。
- 失败。
- 已阻塞。
- 已停止。

状态应在主工作区和顶部轻量展示。

### 证据展示

证据以小缩略图和文本摘要展示。

证据类型：

- 页面截图。
- 接口响应。
- 数据库记录。
- 日志片段。
- DOM 元素信息。

点击证据后，在右侧详情抽屉中查看大图或结构化内容。

### 缺陷草稿

当 AI 判断存在失败时，生成缺陷草稿。

缺陷草稿字段：

- 标题。
- 严重级别。
- 复现步骤。
- 预期结果。
- 实际结果。
- 证据列表。

用户可以：

- 生成缺陷草稿。
- 复制复现步骤。
- 忽略本次失败。

## Agent 与 MCP 数据流

```text
用户输入
  -> Renderer UI
  -> Agent Orchestrator
  -> 意图识别
  -> 测试计划生成
  -> 用户确认
  -> MCP Tool Router
  -> MCP 服务调用
  -> 工具事件流
  -> Renderer UI 展示执行过程
  -> 证据归档
  -> 缺陷草稿生成
```

## Electron 架构约束

Renderer：

- React UI。
- 展示会话、计划、执行流、授权请求和证据。
- 不直接调用 MCP 服务。

Main Process：

- 管理窗口、文件权限、本地配置。
- 承接 Renderer 的安全 IPC 请求。

Agent Runtime：

- 运行 Claude Agent SDK。
- 维护 Agent 会话。
- 将工具调用路由到 MCP 服务。
- 将事件流推送给 Renderer。

MCP Services：

- 按工具边界拆分。
- 每个工具返回结构化结果。
- 工具失败必须返回可展示的错误信息。

## IPC 事件

Renderer -> Main：

- `run:create`
- `run:approve-plan`
- `run:revise-plan`
- `tool:approve`
- `tool:deny`
- `run:stop`

Main -> Renderer：

- `run:created`
- `run:planning`
- `run:plan-ready`
- `run:status-changed`
- `tool:call-started`
- `tool:approval-required`
- `tool:call-completed`
- `tool:call-failed`
- `evidence:created`
- `bug-draft:created`

## 错误处理

Agent 生成计划失败：

- 展示“计划生成失败”。
- 保留用户输入。
- 提供“重试”按钮。

MCP 工具不可用：

- 在工具调用行展示失败状态。
- AI 解释缺失工具对本次测试的影响。
- 允许用户继续、重试或停止。

授权被拒绝：

- 标记工具调用为“已拒绝”。
- AI 尝试使用替代证据。
- 如果无法继续，状态变为“已阻塞”。

执行被停止：

- 停止后续工具调用。
- 保留已完成的执行记录和证据。
- 状态变为“已停止”。

## 成功标准

第一版完成后应满足：

- 测试人员可以用中文输入测试目标。
- UI 能展示 AI 识别、计划、确认、执行、授权、证据、缺陷草稿的完整链路。
- MCP 工具调用过程可追踪。
- 敏感工具调用不会无确认执行。
- 界面保持 Claude Desktop 风格的简洁体验。
- 中文文案自然，不出现英文主导的界面。

## 验收场景

### 场景 1：订单模块测试

输入：

```text
测试订单模块功能
```

期望：

- AI 识别为订单模块测试。
- 生成 4 到 6 个测试计划步骤。
- 用户点击“开始执行”后展示 MCP 工具调用。
- 至少出现一个截图或接口响应证据。

### 场景 2：数据库授权

当 AI 请求调用数据库查询工具时：

- UI 展示“AI 请求查询订单数据库”。
- 用户必须点击“允许”后才继续。
- 拒绝后工具调用标记为“已拒绝”，AI 给出影响说明。

### 场景 3：失败生成缺陷

当订单状态校验失败时：

- 执行步骤标记为失败。
- 右侧抽屉出现缺陷草稿。
- 草稿包含标题、严重级别、复现步骤、预期结果、实际结果和证据。
