审批卡片方案对比 — 简约终端风格

5 种不同方向的终端风格设计，选择一个最接近你预期的，我再精调。所有按钮均可点击交互。

A 极简终端 — 纯文字 + 提示符，无边框 chrome
>approval/request · tool:browser.navigate
打开订单列表页，检查"待支付"状态筛选是否正确。
此操作读取页面 DOM 结构，不修改业务数据。

特点：最干净，只有 > 符号和代码块。适合和对话流无缝融合。

B 深色迷你终端 — 红绿灯 + 命令行，紧凑深底
$ai-assistant request browser.navigate
>打开订单列表页，检查"待支付"状态筛选是否正确。读取页面 DOM 结构，验证筛选控件状态。
工具：browser.navigate
参数：{"url":"/orders?status=pending_payment","waitUntil":"networkidle"}
影响：read-only · 不修改业务数据 · 结果写入会话证据

特点：独立深色面板，和对话区形成视觉对比，终端感最强。

C 浅色系统终端 — 红绿灯标题栏 + 等宽正文，保留暖色调
~/.approval/browser.navigate
$tool:browser.navigate — 需要你审核后才能继续执行
>打开订单列表页，检查"待支付"状态筛选是否正确。读取页面 DOM 结构，验证筛选控件状态。
参数：{"url":"/orders?status=pending_payment","waitUntil":"networkidle"}
影响：read-only · 不修改业务数据 · 结果写入会话证据

特点：暖色系终端，和 Claude Desktop 整体风格统一，有标题栏但不黑暗。

D 行内代码风格 — 无 chrome，对话流内嵌代码块 + 操作
awaitbrowser.navigate// needs approval
({
  url: "/orders?status=pending_payment",
  waitUntil: "networkidle",
  impact: "read-only"
})

特点：最像代码，无任何终端 chrome。适合开发者视角。

E 分栏终端 — 上部工具信息 + 底部状态栏 + 按钮
$tool/approval/browser.navigate
>打开订单列表页，检查"待支付"状态筛选是否正确。读取页面 DOM 结构并验证筛选控件状态。
参数：{"url":"/orders?status=pending_payment","waitUntil":"networkidle"}
影响：read-only · 不修改业务数据 · 结果写入会话证据
waiting_approval

特点：顶部信息 + 底部状态栏，结构清晰分层。
